# ADR-0003: React Native（Expo）の採用

## ステータス

承認済み

## コンテキスト

Otiumアプリケーションは、iOSとAndroidの両方のプラットフォームで動作するモバイルアプリとして開発する必要がありました。クロスプラットフォーム開発のための技術スタックを選択する必要がありました。

## 決定事項

React Native（Expo）を採用します。

## 決定理由

### 1. クロスプラットフォーム開発

- iOSとAndroidの両方で1つのコードベースで開発可能
- 開発コストと時間の削減
- メンテナンスが容易

### 2. Expoの利点

- ネイティブビルドの設定が不要（Expo Goで開発可能）
- 豊富なExpo SDK（画像選択、セキュアストレージなど）
- ホットリロードによる高速な開発体験
- OTA（Over-The-Air）アップデートが可能

### 3. Reactの知識の活用

- Web開発で使用しているReactの知識を活用可能
- コンポーネントベースの開発
- 豊富なエコシステムとライブラリ

### 4. 開発速度

- プロトタイプの作成が迅速
- 開発環境のセットアップが簡単
- コミュニティが大きく、情報が豊富

## 考慮した代替案

### ネイティブ開発（Swift/Kotlin）

**メリット**:
- プラットフォーム固有の機能を最大限に活用可能
- パフォーマンスが最適
- プラットフォームの最新機能をすぐに利用可能

**デメリット**:
- iOSとAndroidで別々のコードベースが必要
- 開発コストと時間が2倍
- メンテナンスが複雑

**結論**: クロスプラットフォーム開発の利点を優先し、React Nativeを選択

### Flutter

**メリット**:
- 高いパフォーマンス
- 統一されたUI/UX
- Googleのサポート

**デメリット**:
- Dart言語の学習コスト
- チームのReact経験を活用できない
- エコシステムがReact Nativeより小さい

**結論**: チームのReact経験を活用できるReact Nativeを選択

### React Native（Bare Workflow）

**メリット**:
- ネイティブモジュールへのアクセスが容易
- より細かい制御が可能

**デメリット**:
- 開発環境のセットアップが複雑
- Expo SDKの利点を活用できない
- ネイティブビルドの知識が必要

**結論**: 現時点ではExpoで十分。将来的にネイティブモジュールが必要になった場合に検討

## 実装詳細

### プロジェクト構成

```
client-app/
├── App.js                 # エントリーポイント
├── screens/              # 画面コンポーネント
├── navigation/           # ナビゲーション設定
├── context/             # React Context
├── api/                 # API通信
└── package.json
```

### 主要な依存関係

- `expo`: ~54.0.25
- `react-native`: 0.81.5
- `@react-navigation/native`: ^7.1.24
- `expo-secure-store`: ~15.0.8
- `expo-image-picker`: ~17.0.10

### 開発フロー

1. `npx expo start` で開発サーバー起動
2. Expo GoアプリでQRコードをスキャン
3. ホットリロードで即座に変更を確認

## 結果

- iOSとAndroidの両方で動作するアプリを1つのコードベースで開発できた
- 開発速度が向上した
- Expo SDKにより、セキュアストレージや画像選択などの機能を簡単に実装できた

## 今後の検討事項

- Expo EAS Buildへの移行（本番ビルド）
- カスタムネイティブモジュールの追加（必要に応じて）
- Bare Workflowへの移行（ネイティブ機能が必要になった場合）

## 制約事項

- Expo Goで動作しないネイティブモジュールは使用できない
- 一部のネイティブ機能はExpo SDK経由でしか利用できない
- カスタムネイティブコードの追加にはEAS Buildが必要
