# ADR-0002: 記録の論理削除の採用

## ステータス

承認済み

## コンテキスト

Otiumアプリケーションでは、ユーザーが記録を削除できる必要があります。削除方式として、物理削除と論理削除のどちらを採用するか検討する必要がありました。

## 決定事項

記録（records）テーブルでは、論理削除を採用します。

## 決定理由

### 1. データの復元可能性

- ユーザーが誤って削除した場合、データを復元できる
- サポート対応でデータ復元が可能
- ユーザーの信頼性向上

### 2. データの分析・監査

- 削除された記録も含めて分析可能
- ユーザーの行動パターンを分析できる
- 監査ログとして機能

### 3. 実装の簡潔性

- 物理削除と比較して、実装が簡単
- 外部キー制約の影響を受けにくい
- データベースの整合性を保ちやすい

### 4. パフォーマンス

- 削除処理が高速（UPDATEのみ）
- インデックスを活用して、有効な記録のみを取得可能

## 考慮した代替案

### 物理削除

**メリット**:
- データベースの容量を節約できる
- クエリがシンプル（WHERE句が不要）

**デメリット**:
- データの復元が不可能
- 外部キー制約の影響を受ける可能性
- ユーザーの誤操作によるデータ損失のリスク

**結論**: ユーザーのデータ保護を優先し、論理削除を選択

### アーカイブテーブルへの移動

**メリット**:
- メインテーブルのサイズを小さく保てる
- クエリパフォーマンスが向上する可能性

**デメリット**:
- 実装が複雑
- データの整合性管理が複雑
- 現時点では記録数が少なく、パフォーマンス問題は発生していない

**結論**: 現時点では不要。将来的に記録数が増えた場合に検討

## 実装詳細

### テーブル設計

```sql
CREATE TABLE records (
  id INT PRIMARY KEY AUTO_INCREMENT,
  user_id INT NOT NULL,
  title VARCHAR(255) NOT NULL,
  description TEXT,
  date_logged DATE NOT NULL,
  invalidation_flag TINYINT(1) DEFAULT 0,
  delete_at TIMESTAMP NULL,
  -- その他のカラム...
);
```

### 削除処理

```javascript
// 論理削除
UPDATE records 
SET invalidation_flag = 1, delete_at = NOW() 
WHERE id = ? AND user_id = ?
```

### 取得処理

```javascript
// 有効な記録のみ取得
SELECT * FROM records 
WHERE user_id = ? AND invalidation_flag = 0
ORDER BY created_at DESC
```

### インデックス

```sql
CREATE INDEX idx_user_invalidation ON records(user_id, invalidation_flag);
```

## 結果

- ユーザーのデータ保護が強化された
- 実装が簡潔で保守しやすい
- パフォーマンスへの影響は最小限

## 今後の検討事項

- 論理削除された記録の物理削除（一定期間後に自動削除）
- 削除された記録の復元機能（UI実装）
- アーカイブテーブルへの移動（記録数が増えた場合）

## 補足: カテゴリーの物理削除

カテゴリー（categories）テーブルでは、物理削除を採用しています。

**理由**:
- カテゴリーは記録と比較して重要度が低い
- カテゴリー削除時、関連する記録の `category_id` を NULL にするだけで十分
- シンプルな実装を優先
