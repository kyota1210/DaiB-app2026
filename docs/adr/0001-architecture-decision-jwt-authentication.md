# ADR-0001: JWT認証方式の採用

## ステータス

承認済み

## コンテキスト

Otiumアプリケーションでは、ユーザー認証が必要です。モバイルアプリとREST APIの間で、セキュアでスケーラブルな認証方式を選択する必要がありました。

## 決定事項

JWT（JSON Web Token）ベースの認証方式を採用します。

## 決定理由

### 1. ステートレスな認証

- サーバー側でセッション情報を保持する必要がない
- サーバーの水平スケーリングが容易
- サーバー再起動後もトークンが有効

### 2. モバイルアプリとの親和性

- トークンをクライアント側で保存・管理できる
- ネットワーク接続が不安定な環境でも動作
- Expo SecureStoreで安全に保存可能

### 3. 実装の簡潔性

- セッション管理の実装が不要
- トークンの検証が簡単（ミドルウェアで実装可能）
- 標準的なライブラリ（jsonwebtoken）が利用可能

### 4. セキュリティ

- トークンに署名が含まれるため、改ざんを検出可能
- 有効期限を設定できる
- トークンにユーザーIDを含めることで、APIでユーザーを識別可能

## 考慮した代替案

### セッションベース認証

**メリット**:
- サーバー側でセッションを完全に制御可能
- トークンの無効化が容易

**デメリット**:
- サーバー側でセッションストレージが必要
- 水平スケーリング時にセッション共有が必要
- モバイルアプリとの親和性が低い

**結論**: モバイルアプリの特性とスケーラビリティを考慮し、JWTを選択

### OAuth 2.0

**メリット**:
- 標準的な認証プロトコル
- サードパーティ認証（Google、Facebookなど）に対応可能

**デメリット**:
- 実装が複雑
- 現時点では自社認証のみで十分

**結論**: 将来的にサードパーティ認証が必要になった場合に検討

## 実装詳細

### トークンの生成

```javascript
const token = jwt.sign(
  { id: user.id, email: user.email },
  JWT_SECRET,
  { expiresIn: '1d' }
);
```

### トークンの検証

```javascript
// middleware/auth.js
const token = req.headers.authorization?.split(' ')[1];
const decoded = jwt.verify(token, JWT_SECRET);
req.user = decoded;
```

### トークンの保存

- クライアント側: Expo SecureStore（暗号化ストレージ）
- 有効期限: 1日

## 結果

- 実装が簡潔で保守しやすい
- モバイルアプリとの親和性が高い
- スケーラブルな設計

## 今後の検討事項

- リフレッシュトークンの導入（トークンの有効期限を延長）
- トークンの無効化機能（ログアウト時のトークン無効化）
- トークンのローテーション
